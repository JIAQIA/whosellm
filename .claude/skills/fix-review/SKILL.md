# Fix Review

根据 Code Review 报告修复问题。**核心原则**：先验真，再从架构视角修复，最后闭环复核。

输入：$ARGUMENTS

---

## 第一步：解析参数

### 两个参数，均可空

1. **修复模式**（第一个 token）：

| 参数 | 范围 | 说明 |
|------|------|------|
| `block` | 仅阻塞项 | 快速解除合并阻塞 |
| `all`（默认） | 阻塞项 + 建议项 | 完整修复 |
| `discuss` | 全部级别 | 逐条验证+讨论，按严重程度排序 |

2. **报告来源**（剩余部分）：

| 格式 | 说明 |
|------|------|
| 留空 | 从当前对话上下文查找最近的 `/code-review` 输出 |
| `@<文件路径>` | 读取指定文件 |
| `#PR-<N>` | 从 PR review comments 获取：`gh api repos/{owner}/{repo}/pulls/{N}/comments` |

拆分逻辑：第一个 token 匹配 `block`/`all`/`discuss` 则为模式，其余为来源；不匹配则模式默认 `all`，整个输入为来源。

### 解析报告

`/code-review` 输出的报告格式为：

- **阻塞项（必须修复）**：表格中的编号行，类别包括 设计规范 / 测试缺失 / 测试数据 / 风格 / 兼容性 等
- **建议项（推荐改进）**：表格中的编号行，类别包括 DRY / 封装 / 测试缺失 等

提取问题清单后输出解析摘要让用户确认：

```
阻塞项 X 个 / 建议项 Y 个，当前模式（{mode}）将处理 Z 个
```

---

## 第二步：验证问题真实性

**这是最关键的一步。Code Review 基于 diff 片段审查，可能误判——修复不存在的问题比不修复真实问题更危险。**

对待处理范围内的每个问题：

### 2.1 定位与验证

- 读取报告引用的文件和行号的**完整上下文**（不只是 diff 片段）
- 确认代码段当前是否仍存在（可能已被其他改动覆盖）
- 行号偏移时用 Grep 搜索附近上下文定位

### 2.2 判定（三选一）

| 判定 | 含义 | 后续 |
|------|------|------|
| **确认** | 问题真实存在 | 进入第三步 |
| **误判** | Review 理解有误（缺少完整上下文） | 跳过，记录理由 |
| **已修复** | 已被其他改动修复 | 跳过，记录确认 |

### 2.3 输出验证报告

```markdown
| # | 文件:行号 | 类别 | 判定 | 说明 |
|---|----------|------|------|------|
| 1 | xxx.py:42 | 设计规范 | 确认 | 问题存在，Family 归属不正确 |
| 2 | xxx.py:88 | DRY | 误判 | Review 未考虑 ModelFamilyConfig 的继承机制 |
| 3 | test_xxx.py:15 | 测试数据 | 已修复 | 在最近的 commit 中已更正 |

实际需处理：X 个（原 Y 个，排除 Z 个误判/已修复）
```

**使用 AskUserQuestion 等待用户确认后再继续。**

---

## 第三步：设计修复方案（block / all 模式）

**进入 Plan 模式**（EnterPlanMode），从架构视角设计整体方案。

### 3.1 分组归类

- **同文件合并**：同一文件多个问题一次解决
- **因果关系**：某些问题可能是其他问题的症状（如类型错误可能源于配置定义不当）
- **修改顺序**：自底向上（`config.py` / `base.py` → `families/*.py` → `tests/`）

### 3.2 方案设计原则

**禁止的修复方式（补丁）**：
- 在问题点加 `if` 绕过
- 用 `# type: ignore` 消除类型报错
- 用 bare `except:` 吞异常
- 只修报告提到的一处，忽略同类问题
- 复制粘贴而非抽取复用

**要求的修复方式（根治）**：
- 在正确的抽象层级解决问题（利用 `ModelFamilyConfig` 的默认值继承、`SpecificModelConfig` 覆盖机制）
- 多处同类问题一并修复（如同一 Family 下多个模型存在相同的能力配置错误）
- 充分利用项目已有模式（`parse` 模式匹配、`infer_variant_priority()`、Registry Merge 机制等）

### 3.3 输出修复计划

按组输出：根因分析 + 方案 + 涉及文件 + 风险评估。**用户审批后退出 Plan 模式执行。**

---

## 第四步：执行修复

按计划逐组实施：

1. 读取完整文件理解全貌，搜索类似实现作为风格参考
2. 每组修复后简要说明改动
3. 发现计划外关联问题时暂停向用户报告
4. 同步更新受影响的测试

---

## 第五步：验证与复核

### 5.1 质量门禁

```bash
uv run poe qa          # 完整流程：格式化 + 检查 + 类型检查 + 测试
```

如果只想逐步排查失败：

```bash
uv run poe check       # Ruff 检查
uv run poe typecheck   # MyPy 严格模式
uv run poe test        # 单元测试 + 集成测试
```

如果修复涉及特定模型家族，运行针对性测试：

```bash
uv run python -m pytest tests/models/families/test_{family}.py -v
```

所有检查必须通过。如有失败，修复后重新运行。

### 5.2 复核（推荐）

调用 `/code-review` 对本次修复变更进行复核，确认原问题已解决且未引入新问题。

---

## 第六步：讨论模式（discuss）

覆盖**全部级别**（阻塞项 + 建议项 + 通过项中的潜在风险），按严重程度从高到低逐条处理。

**使用 AskUserQuestion 逐条讨论，每次只讨论一个问题：**

1. **验证**：读取代码确认问题是否存在
2. **呈现**：展示代码 + 验证结论 + 影响分析
3. **专业判断**：赞同采纳 / 部分赞同 / 不建议采纳（误判直接建议跳过）
4. **征求意见**：AskUserQuestion 等待用户决策
5. **记录**：采纳 → 待执行 / 拒绝 → 记录理由 / 推迟 → TODO

讨论完成后输出决策摘要表，对「采纳」项按第三步至第五步执行修复。

---

## 第七步：输出修复总结

```markdown
## Fix Review 总结

### 处理结果
| # | 问题 | 验证 | 状态 | 改动文件 |
|---|------|------|------|---------|

### 验证状态
- Ruff 检查：PASS / FAIL
- MyPy 类型检查：PASS / FAIL
- 测试：PASS / FAIL
- 复核：通过 / 发现 N 个新问题已处理

### 备注
- 误判问题说明（为什么 Review 判断有误）
- 需上下游配合的事项
```

---

## 反模式

| 禁止 | 正确 |
|------|------|
| 不验证就修复 | 每条问题独立验证真实性 |
| 逐条打补丁 | 先全局分析，分组归类修复 |
| 问题不存在也硬改 | 标记误判，记录理由跳过 |
| 修复引入 `# type: ignore` / bare `except:` | 修复不降低代码质量 |
| 只修报告提到的一处 | 同类问题一并排查修复 |
| 跳过用户确认 | 验证结果和修复计划都等审批 |
| discuss 模式不用 AskUserQuestion | 必须逐条交互 |
